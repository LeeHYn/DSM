# DSM (Daily Schedule Managements) 시스템 아키텍처 명세서 (v1.3)

> **v1.2 → v1.3 주요 변경 사항**
> - **프레임워크 전면 개편**: Next.js (서버리스) → **NestJS (Node.js VPS/PaaS)**
> - **인증 플로우 개편**: NextAuth.js → **자체 커스텀 JWT Auth (Passport.js 등 활용)**
> - **스케줄러 개편**: 외부 QStash 훅 → **NestJS 내부 `node-cron`** (또는 BullMQ)
> - **통신 프로토콜 추가**: HTTP RESTful API 외에 **WebSocket(Socket.io)** 실시간 양방향 통신 추가
> - Vercel 호스팅에서 Render, AWS ECS 등 컨테이너 기반 호스팅으로 변경

## 1. 문서 개요
본 문서는 DSM 모바일 애플리케이션의 신뢰성 있는 일과 알림과 실시간 랭킹 동기화를 위해 확정된 프론트엔드, 백엔드, 데이터베이스 및 주요 인프라의 아키텍처를 정의합니다. v1.3에서는 서버리스의 제약(WebSocket 불가능, 백그라운드 태스크 제약)을 벗어나 **NestJS 기반의 독립 서버**로 아키텍처를 전환했습니다.

## 2. 핵심 기술 스택 및 채택 사유

### 2.1 클라이언트 / 백엔드 / DB

| 구분 | 기술 스택 | 주요 역할 | 채택 사유 |
| :--- | :--- | :--- | :--- |
| **프론트엔드 (Client)** | **React Native** | iOS/Android 모바일 앱 UI/UX 구현 | - 단일 코드베이스로 양대 OS 동시 지원 |
| **백엔드 (API Server)** | **NestJS (Node.js)** | API 라우트, WebSocket 유지, 백그라운드 스케줄러 처리 | - 체계적인 모듈형 아키텍처 제공<br>- WebSocket 및 내부 Cron 잡 네이티브 지원<br>- 콜드 스타트 지연 없음 |
| **데이터베이스** | **PostgreSQL** (Supabase 또는 Neon) | 사용자, 일과, 점수, 랭킹, **FCM 토큰** 저장 | - 윈도우 함수로 랭킹 백분위 산출 용이<br>- 강력한 데이터 무결성 보장 |
| **ORM** | **Prisma** (또는 TypeORM) | 타입 안전한 DB 액세스 및 마이그레이션 관리 | - TypeScript 친화적 |

### 2.2 인증·캐싱·알림·스토리지

| 구분 | 기술 스택 | 주요 역할 | 채택 사유 |
| :--- | :--- | :--- | :--- |
| **인증** | **자체 JWT Auth** | 소셜 로그인 처리 및 모바일 앱 세션 관리 | - NextAuth의 제약에서 벗어나 모바일에 최적화된 Access/Refresh 토큰 발급 및 관리 |
| **실시간 통신** | **WebSocket (Socket.io)** | 랭킹 및 데이터 갱신 실시간 브로드캐스트 | - API 폴링(Polling) 오버헤드 제거 |
| **캐싱** | **Redis (Upstash 또는 내장)** | 랭킹 데이터 캐싱, 세션 보조 | - DB 쿼리 부하 분산 |
| **알림 (Push)** | **FCM (Firebase Cloud Messaging)** | 정각 알림 발송 | - 완전 무료, iOS/Android 통합 채널 |
| **스케줄러** | **NestJS `@nestjs/schedule` (node-cron)** | 일과 정각에 FCM 발송 트리거 | - 별도 외부 스케줄러 불필요, 백엔드 내부 로직과 완벽 통합 |
| **이미지 스토리지** | **Cloudflare R2** 또는 **Supabase Storage** | 프로필 이미지 업로드/제공 | - 저렴하거나 무료인 클라우드 스토리지 |

## 4. 주요 데이터 흐름

### 4.1 일과 등록 및 알림 예약 흐름 (타임존 통일)
1. 사용자가 클라이언트(React Native)에서 일과를 등록 (예: KST 오후 3시)
2. 클라이언트는 이를 **UTC 타임스탬프(06:00 UTC)** 로 변환하여 NestJS API에 전송
3. NestJS는 PostgreSQL에 레코드 저장 후, 내부 메모리(또는 Redis/BullMQ)의 스케줄러(Cron) 큐에 **06:00 UTC** 발송 작업 등록
4. 클라이언트 로컬 타이머에도 동일 일정 등록 (인앱 즉시 알림용)

### 4.2 정각 알림 발송 흐름 (백그라운드/종료 상태)
1. UTC 기준 정각 도달 시 NestJS 내부 스케줄러가 작업 트리거
2. DB에서 사용자 `fcm_token` 조회 후 FCM Admin SDK를 이용해 발송 요청
3. FCM이 사용자 기기로 OS 푸시 알림 전달

### 4.3 정각 알림 발송 흐름 (포그라운드 상태)
1. 클라이언트 로컬 타이머가 1초 단위로 체크하여 정각 도달 시 인앱 팝업/사운드 즉시 트리거
2. WebSocket이 연결되어 있다면, 서버에서 발생한 스케줄러 이벤트가 WS를 통해서도 브로드캐스트됨 (중복 알림 방지 로직 적용)

### 4.4 오프라인 및 충돌 동기화 흐름
1. 오프라인 상태에서 사용자가 일과 체크 시 로컬 SQLite에 임시 저장되고 `updated_at` 갱신
2. 네트워크 연결 시 백그라운드 큐를 통해 NestJS로 일괄 전송
3. 서버는 전달받은 데이터의 `updated_at`과 DB의 `updated_at`을 비교하여 **최신 데이터(Last-Write-Wins)**만 병합

## 6. 보안 및 운영 정책
* **API 인증:** JWT Access Token(짧은 만료) 및 HttpOnly Cookie/Secure Storage 기반 Refresh Token 사용
* **타임존 원칙:** 백엔드/DB의 모든 날짜 연산 및 스케줄링은 **UTC 기준**을 철저히 준수
* **FCM 보안:** 서버 측에 Firebase Admin SDK 비공개 키를 환경변수로 관리하여 발송 권한 통제
