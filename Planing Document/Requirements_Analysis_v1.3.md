# DSM (Daily Schedule Managements) 요구사항 분석서 (v1.3)

> **v1.2 → v1.3 주요 변경 사항**
> - **[FR-01]** NextAuth.js → 자체 JWT Auth 기반으로 인증 플로우 재설계
> - **[FR-05, FR-09]** Upstash QStash 제거 및 NestJS 내부 스케줄러(node-cron) 사용, FCM 토큰 DB 연동 구체화
> - **[NFR-02, NFR-03]** WebSocket 양방향 통신 추가 및 실시간 랭킹 변동 반영
> - **[NFR-05]** 오프라인 충돌 해결 시 타임스탬프(`updated_at`) 우선 정책 추가
> - **[제약사항]** 타임존 처리(UTC 통일) 정책 명시 및 서버리스 제약 문구 삭제

## 1. 프로젝트 개요
* **프로젝트명:** DSM (Daily Schedule Managements)
* **구분:** 모바일 애플리케이션 (iOS / Android, React Native)
* **시스템 목적:** 사용자의 일과 계획 및 실행 내역을 관리하고, 달성률 기반의 점수 산출 및 랭킹·티어 시스템을 통해 객관적인 성취도 지표를 제공함.

---

## 2. 사용자 대상 및 주요 목표
* **주 사용층:** 정해진 일과 및 일정을 체계적으로 관리하고 기록하고자 하는 스마트폰 사용자.
* **주요 목표:**
  * 직관적인 일정 입력 및 관리 기능 제공
  * 점수·백분위·티어 데이터를 통한 사용자 성취도 시각화
  * FCM 푸시 + WebSocket + 로컬 타이머 결합으로 신뢰성·즉시성 양립한 정각 알림 제공

---

## 3. 기능 요구사항 (Functional Requirements)

| ID | 요구사항명 | 상세 설명 | 중요도 |
| :--- | :--- | :--- | :--- |
| **FR-01** | **계정 및 프로필 관리 (JWT)** | - 소셜 로그인(Google/Kakao/Apple) 기반 회원가입·로그인·로그아웃<br>- 닉네임 및 프로필 이미지 설정/수정<br>- **NestJS 백엔드에서 자체 발급하는 JWT(Access/Refresh Token) 기반 인증 인가** | 상 |
| **FR-02** | **일과 관리 (CRUD)** | - 날짜별 일과 추가/조회/수정/삭제<br>- 각 일과의 시작·종료 시간 설정 (클라이언트 전송 시 **UTC 기준 통일**)<br>- Optimistic UI 적용으로 즉시 반영 | 상 |
| **FR-03** | **점수 산출 시스템** | - 공식: `Σ(완료한 일과의 난이도 점수) × 달성률 보정 계수`<br>- 난이도 점수: 하(10)/중(20)/상(30)<br>- 달성률 보정: 100%→×1.5, 80%이상→×1.3, 60%이상→×1.0, 60%미만→×0.7<br>- **[어뷰징 방지] 1일 최대 획득 점수 상한 및 일과 등록 개수(최대 20개) 제한** | 상 |
| **FR-04** | **랭킹 및 백분위 조회** | - **일간 랭킹**: 당일 점수 기준 (UTC 자정 기준 초기화)<br>- **주간/누적 랭킹**: 최근 7일 및 가입 이후 전체 누적 점수 기준<br>- 각 랭킹별 본인 등수 및 상위 백분위(%) 제공 | 상 |
| **FR-05** | **하이브리드 정각 알림** | - **백그라운드/종료 상태**: 백엔드(NestJS) 내부의 **`node-cron`** 스케줄러가 정각에 FCM 발송 트리거<br>- **포그라운드 상태**: **WebSocket** 알림 수신 및 로컬 타이머를 결합하여 인앱 팝업/사운드 즉시 트리거 | 상 |
| **FR-06** | **캘린더 뷰** | - 월간 및 주간 단위의 달력 UI 제공<br>- 날짜별 일과 등록 상태 및 달성률 인디케이터 표시 | 중 |
| **FR-07** | **일과 카테고리 관리** | - 일과 등록 시 카테고리 및 난이도 선택<br>- 카테고리별 통계 화면 제공 | 중 |
| **FR-08** | **티어 시스템** | - 누적 점수 기준 6단계 티어(브론즈→마스터)<br>- 승급/강등 시 인앱 알림 | 중 |
| **FR-09** | **FCM 토큰 DB 연동** | - 앱 실행 및 로그인 성공 시 클라이언트가 FCM 디바이스 토큰을 백엔드로 전송<br>- 서버는 사용자 DB 테이블의 `fcm_token` 필드를 최신 상태로 유지하며 로그아웃 시 무효화 처리 | 상 |

## 4. 비기능 요구사항 (Non-Functional Requirements)

| ID | 요구사항명 | 상세 설명 |
| :--- | :--- | :--- |
| **NFR-01** | **알림 신뢰성 및 타임존** | - **타임존(Timezone)**: 서버와 DB는 UTC 기준으로 동작하며, 예약 알림도 UTC 타임스탬프 기준으로 `node-cron`에 등록하여 오작동을 방지함<br>- 최초 진입 시 OS 알림 권한 요청 |
| **NFR-02** | **WebSocket 동기화** | - **WebSocket (Socket.io 등)**을 사용하여 서버-클라이언트 간 양방향 통신 채널 유지<br>- 사용자의 점수 변동이나 랭킹 변동 발생 시 접속 중인 클라이언트에 즉시 이벤트 브로드캐스트 |
| **NFR-03** | **시스템 성능 (서버)** | - 랭킹 계산은 배치 작업으로 수행하되, WebSocket을 통해 캐싱된 Redis 데이터를 즉시 푸시<br>- PostgreSQL의 윈도우 함수를 활용하여 백분위 산출 최적화 |
| **NFR-04** | **사용성 (UI/UX)** | - 일과 등록 단계 최소화(3 탭 이내 완료)<br>- 점수·랭킹·티어 데이터의 명확한 시각적 제공 |
| **NFR-05** | **오프라인 충돌 해결** | - 네트워크 연결 끊김 시 일과 CRUD는 로컬(AsyncStorage/SQLite) 임시 저장<br>- 네트워크 복구 시 자동 동기화. **데이터 충돌 시 레코드의 `updated_at` 타임스탬프가 더 최근인 데이터를 우선 반영(Last-Write-Wins)** |
| **NFR-06** | **확장성** | - 컨테이너 기반(Docker) 배포로 VPS 또는 클라우드(Render, AWS ECS 등) 호스팅을 통해 트래픽 증가 대응 가능 |

---

## 5. 제약사항 (Constraints)
* **네트워크 의존성:** 실시간 랭킹 산출, FCM 알림 예약, WebSocket 통신은 네트워크가 연결된 상태에서 작동합니다. 
* **OS 알림 제약:** iOS는 사용자가 알림 권한을 거부할 경우 FCM 푸시가 전달되지 않으므로, 인앱 로컬 타이머만으로 동작해야 합니다.
* **타임존 통일 엄수:** 모든 클라이언트는 디바이스 로컬 시간을 기반으로 서버와 통신할 때 **반드시 ISO 8601 형식의 UTC 타임스탬프**를 사용해야 하며, 화면 표시 시에만 로컬 타임존(예: KST)으로 변환해야 합니다.
