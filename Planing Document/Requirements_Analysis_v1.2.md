# DSM (Daily Schedule Managements) 요구사항 분석서 (v1.2)

> **v1.1 → v1.2 주요 변경 사항**
> - FR-05를 WebSocket → FCM 푸시 + 로컬 타이머 하이브리드로 재정의
> - FR-04에 랭킹 산출 주기(일/주/누적) 명시
> - FR-07 (일과 카테고리·난이도 관리) 신규 추가
> - FR-08 (티어 시스템) 신규 추가
> - NFR-01 (알림 신뢰성) 구체화 및 NFR-05 (오프라인 동기화) 신규 추가

## 1. 프로젝트 개요
* **프로젝트명:** DSM (Daily Schedule Managements)
* **구분:** 모바일 애플리케이션 (iOS / Android, React Native)
* **시스템 목적:** 사용자의 일과 계획 및 실행 내역을 관리하고, 달성률 기반의 점수 산출 및 랭킹·티어 시스템을 통해 객관적인 성취도 지표를 제공함.

---

## 2. 사용자 대상 및 주요 목표
* **주 사용층:** 정해진 일과 및 일정을 체계적으로 관리하고 기록하고자 하는 스마트폰 사용자.
* **주요 목표:**
  * 직관적인 일정 입력 및 관리 기능 제공
  * 점수·백분위·티어 데이터를 통한 사용자 성취도 시각화
  * FCM 푸시 + 로컬 타이머 결합으로 신뢰성·즉시성 양립한 정각 알림 제공

---

## 3. 기능 요구사항 (Functional Requirements)

| ID | 요구사항명 | 상세 설명 | 중요도 |
| :--- | :--- | :--- | :--- |
| **FR-01** | **계정 및 프로필 관리** | - 소셜 로그인(Google/Kakao/Apple) 기반 회원가입·로그인·로그아웃<br>- 닉네임 및 프로필 이미지 설정/수정<br>- NextAuth.js 기반 세션 관리 | 상 |
| **FR-02** | **일과 관리 (CRUD)** | - 날짜별 일과 추가/조회/수정/삭제<br>- 각 일과의 시작·종료 시간 설정<br>- Optimistic UI 적용으로 즉시 반영 | 상 |
| **FR-03** | **점수 산출 시스템** | - 공식: `Σ(완료한 일과의 난이도 점수) × 달성률 보정 계수`<br>- 난이도 점수: 하(10)/중(20)/상(30)<br>- 달성률 보정: 100%→×1.5, 80%이상→×1.3, 60%이상→×1.0, 60%미만→×0.7<br>- 일/주/월 단위 점수 누적·저장 | 상 |
| **FR-04** | **랭킹 및 백분위 조회** | - **일간 랭킹**: 당일 점수 기준 (자정 기준 초기화)<br>- **주간 랭킹**: 최근 7일 누적 점수 기준<br>- **누적 랭킹**: 가입 이후 전체 누적 점수 기준<br>- 각 랭킹별 본인 등수 및 상위 백분위(%) 제공 | 상 |
| **FR-05** | **하이브리드 정각 알림** | - **백그라운드/종료 상태**: 서버 스케줄러(Upstash QStash 등)가 정각에 FCM 발송 트리거 → OS 푸시로 전달<br>- **포그라운드 상태**: 클라이언트 로컬 타이머가 1초 단위 체크하여 인앱 팝업/사운드 즉시 트리거<br>- 동일 알림 중복 방지(deduplication) 로직 적용 | 상 |
| **FR-06** | **캘린더 뷰** | - 월간 및 주간 단위의 달력 UI 제공<br>- 날짜별 일과 등록 상태 및 달성률 인디케이터(색상) 표시 | 중 |
| **FR-07** | **일과 카테고리·난이도 관리** | - 일과 등록 시 카테고리(예: 공부/운동/업무/기타) 및 난이도(하/중/상) 선택<br>- 카테고리별 통계 화면 제공 | 중 |
| **FR-08** | **티어 시스템** | - 누적 점수 기준 6단계 티어(브론즈→마스터)<br>- 마이페이지 및 랭킹 화면에 티어 뱃지 노출<br>- 승급/강등 시 인앱 알림 | 중 |

## 4. 비기능 요구사항 (Non-Functional Requirements)

| ID | 요구사항명 | 상세 설명 |
| :--- | :--- | :--- |
| **NFR-01** | **알림 신뢰성 및 즉시성** | - **표준 채널**: 모든 환경에서 FCM이 알림 전달의 표준 채널<br>- **즉시성 보강**: 포그라운드 상태에서는 로컬 타이머가 ms 단위 즉시성 보장<br>- **권한 관리**: 최초 진입 시 OS 알림 권한 요청, 거부 시 인앱 알림만으로 폴백 |
| **NFR-02** | **데이터 동기화** | - 다중 기기 접속 시 사용자 데이터 및 랭킹 정보 동기화<br>- 서버를 단일 소스 오브 트루스(Single Source of Truth)로 활용 |
| **NFR-03** | **시스템 성능 (서버)** | - 랭킹 계산은 10분 주기 배치 작업으로 수행, 결과는 **Redis(Upstash)에 캐싱**<br>- PostgreSQL의 윈도우 함수를 활용하여 백분위 산출 최적화 |
| **NFR-04** | **사용성 (UI/UX)** | - 일과 등록 단계 최소화(3 탭 이내 완료)<br>- 점수·랭킹·티어 데이터의 명확한 시각적 제공 |
| **NFR-05** | **오프라인 동기화** | - 네트워크 미연결 시 일과 CRUD는 로컬 저장소(AsyncStorage/SQLite)에 임시 저장<br>- 네트워크 복구 시 자동 동기화, 충돌 발생 시 서버 데이터 우선 |
| **NFR-06** | **확장성** | - 서버리스(Vercel) 기반으로 트래픽 증가에 따른 자동 스케일링<br>- 1만 DAU까지 무중단 확장 가능한 구조 |

## 5. 제약사항 (Constraints)
* **네트워크 의존성:** 점수 누적, 랭킹 산출, FCM 알림 발송은 네트워크가 연결된 상태에서 작동합니다. 오프라인 입력은 로컬에 임시 저장 후 네트워크 복구 시 동기화됩니다.
* **알림 권한 의존성:** iOS는 사용자가 알림 권한을 거부할 경우 FCM 푸시가 전달되지 않습니다. 이 경우 인앱 로컬 타이머만으로 알림이 제공됩니다.
* **서버리스 환경 제약:** 백엔드가 서버리스(Vercel) 기반이므로 **영속 연결(WebSocket)이 불가**합니다. 따라서 모든 서버 → 클라이언트 메시지는 FCM 또는 클라이언트 측 polling으로 처리합니다.
* **FCM 의존성:** 알림 전달은 Google FCM 서비스에 의존하며, FCM 장애 시 인앱 로컬 알림으로 폴백됩니다.
