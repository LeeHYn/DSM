# DSM (Daily Schedule Managements) 시스템 아키텍처 명세서 (v1.2)

> **v1.1 → v1.2 주요 변경 사항**
> - 누락되었던 인증(NextAuth.js), ORM(Prisma), 캐싱(Redis), 스케줄러(QStash), 스토리지(R2) 정식 편입
> - 알림 데이터 흐름을 FCM 중심으로 명확화
> - 모니터링·로깅 도구(Sentry) 추가
> - 호스팅 환경(Vercel) 명시

## 1. 문서 개요
본 문서는 DSM 프로젝트의 원활한 서비스 운영과 신뢰성 있는 알림 전달을 위해 확정된 프론트엔드, 백엔드, 데이터베이스 및 주요 인프라의 기술 스택과 아키텍처 구조를 정의합니다. v1.2에서는 **WebSocket 기반 양방향 통신을 포기하고, FCM 푸시 + 로컬 타이머 하이브리드** 방식으로 알림 아키텍처가 확정되었습니다.

## 2. 핵심 기술 스택 및 채택 사유

### 2.1 클라이언트 / 백엔드 / DB

| 구분 | 기술 스택 | 주요 역할 | 채택 사유 |
| :--- | :--- | :--- | :--- |
| **프론트엔드 (Client)** | **React Native** | iOS/Android 모바일 앱 UI/UX 구현 | - 단일 코드베이스로 양대 OS 동시 지원<br>- 백엔드와 JS/TS 생태계 공유로 생산성 극대화 |
| **백엔드 (API Server)** | **Next.js (Vercel 호스팅)** | API 라우트 기반 데이터 입출력 및 비즈니스 로직 처리 | - 서버리스 환경으로 자동 스케일링<br>- 트래픽 없을 때 비용 0<br>- API Routes로 빠른 개발 |
| **데이터베이스** | **PostgreSQL (Supabase 또는 Neon)** | 사용자, 일과, 점수, 랭킹 데이터 영구 저장 | - 윈도우 함수로 백분위 산출 용이<br>- 강력한 데이터 무결성 보장<br>- 매니지드 서비스로 운영 부담 절감 |
| **ORM** | **Prisma** | 타입 안전한 DB 액세스 및 마이그레이션 관리 | - TypeScript 친화적<br>- Next.js와의 통합 표준 |

### 2.2 인증·캐싱·알림·스토리지

| 구분 | 기술 스택 | 주요 역할 | 채택 사유 |
| :--- | :--- | :--- | :--- |
| **인증** | **NextAuth.js (Auth.js)** | 소셜 로그인(Google/Kakao/Apple) 및 세션 관리 | - Next.js 표준 인증 라이브러리<br>- JWT 기반 세션으로 서버리스에 적합 |
| **캐싱** | **Redis (Upstash)** | 랭킹 데이터 10분 주기 캐싱, 세션 보조 | - 서버리스 친화적인 HTTP 기반 Redis<br>- 무료 티어로 시작 가능 |
| **알림 (Push)** | **FCM (Firebase Cloud Messaging)** | 정각 알림 발송 (포그라운드/백그라운드/종료 전 상태 모두) | - 완전 무료<br>- iOS·Android 통합 채널<br>- HTTP 기반으로 서버리스 호환 |
| **스케줄러** | **Upstash QStash** | 일과 정각에 FCM 발송 트리거 | - 서버리스 환경의 표준 스케줄러<br>- 메시지 큐 + 지연 실행 지원<br>- 무료 티어 월 500 메시지 (소규모 PoC 가능), 유료 $10/월 |
| **이미지 스토리지** | **Cloudflare R2** 또는 **Supabase Storage** | 프로필 이미지 업로드/제공 | - R2: S3 호환, 송신 비용 0<br>- Supabase: DB와 통합 운영 가능 |

### 2.3 운영·모니터링

| 구분 | 기술 스택 | 주요 역할 |
| :--- | :--- | :--- |
| **에러 추적** | **Sentry** | 프론트엔드/백엔드 예외 추적 및 알림 |

## 4. 주요 데이터 흐름

### 4.1 일과 등록 및 알림 예약 흐름

1. 사용자가 클라이언트(React Native)에서 일과를 등록 (예: 오후 3시 운동)
2. Next.js API가 PostgreSQL에 일과 레코드 저장
3. **동시에 Upstash QStash에 해당 시간(오후 3시) 정각 발송 메시지 예약**
4. 클라이언트 로컬 타이머에도 동일 일정 등록 (인앱 즉시 알림용)

### 4.2 정각 알림 발송 흐름 (백그라운드/종료 상태)

1. 오후 3시 도달 → Upstash QStash가 지정된 Next.js API 엔드포인트(`/api/cron/fire-notification`) 호출
2. API가 해당 일과 정보 조회 후 FCM 발송 요청
3. FCM이 사용자 기기로 OS 푸시 알림 전달
4. 클라이언트 FCM 핸들러가 알림 수신 후 데이터 동기화

### 4.3 정각 알림 발송 흐름 (포그라운드 상태)

1. 클라이언트 로컬 타이머가 1초 단위로 다음 알림 시각 체크
2. 정각 도달 시 인앱 팝업/사운드 즉시 트리거 (ms 단위 즉시성)
3. **중복 방지**: 동일 시각에 FCM도 수신될 경우 클라이언트에서 dedup 처리

### 4.4 점수 산출 및 랭킹 갱신 흐름

1. 사용자가 일과 완료 체크 → Next.js API가 점수 재계산 후 PostgreSQL 업데이트
2. 클라이언트는 응답으로 갱신된 점수 즉시 수신 (Optimistic UI 반영)
3. **랭킹 배치 작업**: 10분 주기로 Vercel Cron이 트리거 → 전체 사용자 점수 기반 랭킹/백분위 산출 → Redis에 캐싱
4. 사용자가 랭킹 탭 진입 시 Redis 캐시에서 즉시 응답

## 6. 보안 및 운영 정책

* **API 인증:** 모든 보호 엔드포인트는 NextAuth.js 세션 토큰(JWT) 검증 필수
* **QStash 트리거 검증:** `/api/cron/*` 엔드포인트는 QStash 서명(Signature) 검증으로 외부 호출 차단
* **민감 정보 관리:** API 키·시크릿은 Vercel 환경변수로 관리, 클라이언트 노출 금지
* **백업:** PostgreSQL 일일 자동 스냅샷 (Supabase/Neon 기본 제공)
* **개인정보 처리:** 회원 탈퇴 시 30일 이내 모든 개인정보 영구 삭제 (GDPR/개인정보보호법 준수)
