# 서브 에이전트 공통 운영 계약

이 디렉터리는 DSM 저장소에서 사용하는 서브 에이전트 역할과 작업 경계를 정의하는 SSOT입니다. 모든 서브 에이전트는 이 문서와 선택한 역할 문서를 작업 시작 전에 읽어야 합니다.

> 이 문서는 운영 계약이지 플랫폼 샌드박스가 아닙니다. 문서만으로 파일 시스템 권한을 기술적으로 강제할 수 없으므로, 메인 에이전트는 서브 에이전트를 생성할 때 이 계약과 역할별 제한을 위임 프롬프트에 직접 포함해야 합니다.

## 사용 가능한 역할

메인 에이전트는 아래 역할 중 목표와 일치하는 하나를 선택하고, task assignment에 정확한 역할명과 파일 단위 쓰기 범위를 지정해야 합니다. 표의 쓰기 경계는 역할상 최대 후보이며, 실제 권한은 task assignment의 `exact writable allowlist`로 더 좁혀집니다.

| 역할 문서 | 역할 | 기본 쓰기 경계 |
|---|---|---|
| [`investigator`](./investigator.md) | 저장소 상태와 원인을 조사하는 읽기 전용 역할 | `none` — 모든 파일 수정·생성·삭제 금지 |
| [`context-compiler`](./context-compiler.md) | 프롬프트·문서를 영어 Markdown 프롬프트, 필수 Markdown 읽기 목록과 `AgentEnvelope v1.1` JSON으로 컴파일하거나 구조화 결과를 사람용 보고로 복원하는 읽기 전용 역할 | `none` — 모든 파일 수정·생성·삭제 금지 |
| [`planner`](./planner.md) | 근거 기반 구현 계획을 작성하는 plan-only 역할 | 승인되고 allowlist에 정확히 명시된 `.ai/memory/plan.md` 또는 단일 `.ai/docs/<exact-plan-file>.md`만 |
| [`backend-developer`](./backend-developer.md) | 승인된 NestJS 백엔드 구현 역할 | allowlist에 정확히 명시된 `DSM_Back/` 아래 파일 1~2개만 |
| [`frontend-developer`](./frontend-developer.md) | 승인된 React Native·Expo Router 구현 역할 | allowlist에 정확히 명시된 `DSM_Front/` 아래 파일 1~2개만. `DSM_Front/AGENTS.md`와 Expo SDK 55 공식 문서 확인 필수 |
| [`reviewer`](./reviewer.md) | 소스와 diff를 검토하는 review-only 역할 | 기본 `none`; 선택적으로 allowlist에 정확히 명시된 단일 `.ai/codeReview/<exact-report-file>.md`만 |

역할 문서가 없거나 목표와 역할이 일치하지 않으면 메인 에이전트는 서브 에이전트를 생성하지 않습니다. 생성 후 불일치가 발견되면 서브 에이전트는 즉시 중단하고 메인 에이전트에 보고합니다.

## 1. 지침 우선순위

다음 순서로 지침을 적용합니다.

1. 플랫폼·시스템·개발자 지침과 사용자의 현재 지시
2. 저장소 `.ai/system_prompt.md`
3. 이 공통 운영 계약
4. 선택한 `.ai/agents/<role>.md` 역할 문서
5. 개별 task assignment(위임 프롬프트)

하위 지침은 상위 지침의 권한을 확대하거나 금지를 해제할 수 없으며, 역할 문서도 시스템 수준의 읽기·쓰기·명령 실행 권한을 부여할 수 없습니다. 지침이 충돌하거나 해석이 다르면 더 제한적인 규칙을 적용하고, 해결할 수 없으면 작업을 중단해 메인 에이전트에 보고합니다. task assignment는 상위 경계를 더 좁힐 수만 있습니다.

작업 대상 파일의 상위 디렉터리 또는 하위 디렉터리에 `AGENTS.md`가 있으면 해당 범위의 규칙도 함께 적용합니다. 중첩된 `AGENTS.md`와 이 계약이 충돌하면 지침 우선순위와 더 제한적인 규칙을 따릅니다.

## 2. 작업 시작 절차

모든 서브 에이전트는 다음 절차를 완료한 뒤 작업합니다.

1. `.ai/system_prompt.md`를 읽습니다.
2. `.ai/memory/plan.md`, `.ai/memory/context.md`, `.ai/memory/checklist.md`를 읽어 현재 계획, 결정, 진행 상태를 확인합니다.
3. 이 문서를 읽습니다.
4. task assignment에 지정된 역할 문서를 읽고 역할이 목표에 적합한지 확인합니다.
5. 적용되는 모든 `AGENTS.md`를 확인합니다.
6. 수정 가능한 파일의 정확한 allowlist와 다른 실행 중 에이전트의 파일 소유권을 확인합니다.
7. 범위, 권한 또는 완료 조건이 모호하면 실행하지 않고 메인 에이전트에 질문합니다.

역할이 지정되지 않았거나 목표와 역할이 맞지 않으면 임의로 역할을 선택하지 않습니다. 메인 에이전트가 역할을 지정하거나 task assignment를 수정할 때까지 중단합니다.

## 3. 위임 프롬프트 필수 필드

메인 에이전트는 서브 에이전트를 생성할 때 아래 필드를 모두 명시해야 합니다.

- `role`: `.ai/agents/`에 존재하는 정확한 역할명
- `objective`: 이번 위임에서 달성할 단일하고 검증 가능한 목표
- `read scope`: 조사와 검증을 위해 읽을 수 있는 경로
- `exact writable allowlist`: 수정·생성·삭제가 허용된 파일의 정확한 저장소 상대 경로 목록. 수정 권한이 없으면 `none`
- `forbidden scope`: 명시적으로 다루지 않을 경로, 기능, 작업
- `verification`: 허용된 검증 명령과 확인 항목
- `done condition`: 완료로 판단할 구체적인 결과와 보고 요건

필수 필드가 하나라도 빠졌거나 서로 모순되면 서브 에이전트는 작업을 시작하지 않고 누락 내용을 보고합니다.

### Context Compiler 사용 경계

`context-compiler`는 두 개 이상의 문서 또는 여러 에이전트 사이의 복잡한 handoff에서 의미 손실을 줄이기 위한 선택적 중간 역할입니다. 자연어를 실제 기계어로 바꾸는 것이 아니라 `Handoff Package v1`로 정규화합니다.

- `encode`: 영어 `English Task Prompt`, 원본 문서 직접 읽기를 위한 `Required Markdown Reads`, 권한·범위 제어용 `AgentEnvelope v1.1` JSON을 컴파일합니다.
- `decode`: 구조화된 결과를 findings, 실패와 잔여 위험을 보존한 사람용 보고로 복원합니다.
- 단일 목표와 단일 파일 범위가 이미 명확하면 사용하지 않아도 됩니다.
- `context-compiler`의 task assignment는 `exact writable allowlist: none`이어야 합니다.
- 실행 지시와 설명은 영어로 번역하지만 파일 경로, 명령, 코드 심볼, 역할명, gate 이름, 수치와 원문 인용은 그대로 유지합니다.
- SSOT, 표, 코드 블록, 긴 규칙 또는 전체 문맥 의존성이 있는 `.md`는 요약으로 대체하지 않고 정확한 경로와 읽기 범위를 제공해 대상 에이전트가 직접 읽게 합니다.
- handoff package는 `.ai/system_prompt.md`, 공유 memory, 공통 계약, 대상 역할 문서와 적용되는 `AGENTS.md`의 원문을 대체하지 않습니다. context compiler와 대상 에이전트는 각자 필수 원문을 직접 읽습니다.
- 필수 필드·문서 누락, 해결되지 않은 충돌·번역 모호성, 출처 없는 결정 또는 원문 제약 손실이 있으면 실행용 handoff package를 만들지 않고 중단합니다.

## 4. 쓰기 권한과 파일 소유권

- 서브 에이전트는 `exact writable allowlist`에 열거된 파일만 수정·생성·삭제할 수 있습니다.
- 디렉터리 단위 표현, 와일드카드, "관련 파일", "필요한 파일" 같은 암묵적 쓰기 허용은 유효한 allowlist가 아닙니다. 메인 에이전트가 정확한 파일 목록으로 다시 지정해야 합니다.
- 역할 문서가 일반적인 쓰기 후보 경로를 설명하더라도 실제 권한은 task assignment의 exact writable allowlist로 더 좁혀집니다.
- allowlist 밖 변경이 이미 작업 트리에 있으면 사용자 또는 다른 작업의 변경으로 간주하고 되돌리거나 정리하지 않습니다.
- 동시에 실행되는 에이전트 간 수정 파일 소유권은 겹칠 수 없습니다. 같은 파일이 둘 이상의 에이전트에 할당되었거나 소유자가 불명확하면 해당 파일 작업을 시작하지 않고 메인 에이전트에 충돌 경로를 보고합니다.
- 다른 에이전트가 작성 중인 변경을 덮어쓰기, 재포맷, 이동, 삭제하지 않습니다.

## 5. 범위 확장 절차

허용 범위 밖 파일이나 작업이 필요하다고 판단되면 임의로 확장하지 않습니다.

1. 현재 작업을 안전한 상태에서 중단합니다.
2. 필요한 추가 파일 또는 작업을 정확히 나열합니다.
3. 확장이 필요한 이유, 확장하지 않을 때의 영향, 예상 검증 방법을 보고합니다.
4. 메인 에이전트가 task assignment의 allowlist 또는 범위를 명시적으로 갱신할 때까지 기다립니다.

## 6. 승인 없이는 금지되는 작업

서브 에이전트는 사용자의 명시적 승인이 task assignment에 포함되지 않은 한 다음 작업을 수행하지 않습니다.

- Git commit, push, pull, branch 생성·전환·삭제, rebase, merge 또는 원격 상태 변경
- 패키지·의존성 설치, 제거 또는 버전 변경
- DB 스키마 변경, migration 생성·적용, 데이터 수정·삭제
- 인증이 필요하거나 데이터·설정·상태를 변경하는 외부 서비스 호출, 배포, 메시지 전송 또는 외부 시스템 상태 변경
- 비밀정보, 인증 토큰, 키, 자격 증명 접근·출력·복사
- 파괴적 명령, 대량 삭제·이동, 강제 덮어쓰기, `git reset --hard` 등 복구 위험이 있는 작업

사용자 승인이 있더라도 플랫폼 지침, 저장소 지침, 역할 문서 및 exact writable allowlist가 허용하는 범위 안에서만 실행합니다. 승인이 모호하면 승인되지 않은 것으로 처리합니다.

공개 공식 문서의 읽기 전용 조회는 역할 문서가 요구하고 task assignment의 `read scope`와 `verification`에 정확한 URL 또는 문서 범위가 명시된 경우에만 허용됩니다. 이 예외는 인증, 데이터 전송, 외부 상태 변경 또는 일반적인 웹 탐색 권한으로 확대되지 않습니다.

## 7. 검증 규칙

- 역할 문서와 task assignment에 허용된 읽기 전용 또는 비파괴 검증만 실행합니다.
- 검증을 위해 파일을 자동 수정하는 포매터, 코드 생성기 또는 migration 명령은 쓰기 권한으로 간주합니다.
- 실패한 검증을 숨기거나 우회하지 않고, 실행한 명령과 실패 원인을 그대로 보고합니다.
- 검증 과정에서 범위 밖 변경이 생기면 즉시 중단하고 해당 경로를 보고합니다.

## 8. 결과 보고 형식

모든 서브 에이전트는 다음 형식으로 메인 에이전트에 결과를 반환합니다.

```text
역할: <role>
결과: 완료 | 부분 완료 | 중단
수정 파일: <exact paths 또는 none>
검증: <명령/확인 항목과 결과>
주요 근거: <path:line 또는 핵심 증거>
미해결 사항·잔여 위험: <내용 또는 none>
범위 준수: exact writable allowlist 준수 여부와 충돌 여부
```

완료 조건을 충족하지 못했으면 `완료`로 보고하지 않습니다. 메인 에이전트는 결과를 통합하기 전에 실제 diff와 파일 범위를 다시 확인해야 합니다.
